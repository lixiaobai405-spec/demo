# 美太 AI 商业创新智能体推进计划

## 1. 文档目的

本文档基于以下三部分信息形成：

- 第一版指导书：`美太AI商业创新智能体_PRD一页摘要.md`
- 第二版指导书：`V2美太AI商业创新智能体功能蓝图_总结.md`
- 当前项目代码与现状文档：`docs/PROJECT_SUMMARY_CN.md`

目标是明确：

1. 当前项目已经完成了什么
2. 与 V1 / V2 目标相比还缺什么

4. 每个阶段应交付什么结果

本文档定位为“项目推进计划”，强调可执行性，而不是泛泛的产品描述。

## 2. V1 与 V2 指导方向总结

## 2.1 V1 的核心要求

第一版指导书强调的是一个**课程陪伴式、低门槛、强产出**的智能体，核心是围绕商业模式画布与“道-术-为”方法论，帮助非技术管理者在课程中完成：

- 企业诊断
- AI 提效场景筛选
- 差异化策略设计
- AI 商业创新落地规划
- Word / PDF 导出
- 课后 30 天跟进

V1 的重点在于：

- 用户不需要懂提示词，也能被引导完成产出
- 最终交付物要能拿去内部汇报和推进
- 课程现场要稳定、可控、可溯源

## 2.2 V2 的核心强化

第二版指导书在保留 V1 主线的基础上，做了几个关键升级：

### 2.2.1 前置准备更明确

- 更强调课前输入文档
- 要求学员围绕“某一具体业务”提前准备画布现状与痛点
- 系统需要更强的非结构化输入解析与标准化能力

### 2.2.2 点阶段更细化

V2 不再只是“推荐场景”，而是强调：
2
- 先从 9 要素中选出 2-3 个最想突破的要素
- 再围绕突破要素延展多个发展方向
- 最后再做场景优先级评估

这意味着系统需要从“直接给结果”升级为“引导用户共创突破路径”。

### 2.2.3 线阶段目标更具体

V2 对“差异化竞争力设计”要求更高，强调：

- 将多个点状能力串联成系统性方案
- 重构价值主张（VP）
- 形成更像咨询交付物的竞争力提升路径

### 2.2.4 面阶段新增商业终局导向

V2 明确提出最终规划不只是路线图，而要体现：

- 私域
- 生态
- OPC

也就是要从“单公司内部优化”进一步走向“商业终局设计”。

## 3. 当前项目现状判断

结合当前仓库代码，项目已完成一个较完整的 Demo 底座，主要已具备以下能力：

- Assessment 主流程
- 企业画像生成
- 商业模式画布诊断
- Top 场景推荐
- 案例匹配
- 结构化报告生成
- Markdown / DOCX / 打印版导出
- LLM 报告增强模式
- RAG 模块基础接入

换句话说，项目当前已经具备 **“从输入到报告交付”的基础闭环**。

但若以 V1 / V2 的指导书为目标，当前系统仍然主要停留在：

- “可运行 Demo”
- “结构化结果生成器”
- “规则 + 模板驱动的基础咨询工具”

距离 V2 所要求的“课程陪伴式、分阶段引导式、方法论驱动、可持续跟进”的产品形态，还有明显差距。

## 4. 差距分析

## 4.1 已基本对齐的部分

以下能力与指导书目标已经较为接近：

- 已有从画像到报告的主流程闭环
- 已支持商业画布诊断、场景推荐、案例匹配、报告生成
- 已有导出能力，满足报告交付基础要求
- 已有流程状态恢复与持久化，具备继续做“上下文记忆”的基础
- 已接入 LLM 和 RAG 的扩展接口，具备增强空间

## 4.2 尚未充分对齐的部分

以下是当前推进的重点缺口。

### 4.2.1 缺少“课前输入文档 + 初始化导入”能力

V2 强调学员在课前围绕某一具体业务准备材料，但当前系统主要仍然依赖页面问卷录入，尚未体现：

- 课前输入模板
- 文档导入
- 非结构化文本解析
- 自动预填画布格子

### 4.2.2 缺少“突破要素选择 + 方向延展”交互

当前系统的场景推荐以规则筛选 Top 场景为主，还没有形成 V2 要求的关键过程：

- 选 2-3 个突破要素
- 对突破要素展开多个创新方向
- 让用户选择方向，再生成场景矩阵

这部分是从“自动生成”向“共同设计”升级的核心。

### 4.2.3 缺少完整的“线”阶段模块

当前报告中虽然已有“差异化竞争力设计”章节，但更偏输出结果，而不是完整的独立阶段能力。仍缺：

- VP 重构逻辑
- 系统串联型场景建模
- 一客一策 / 智能运营 / 柔性供应链等能力框架化
- 可视化展示“点如何串成线”

### 4.2.4 缺少“商业终局”模型

V2 的“面”阶段要求输出商业终局模型，当前项目尚未看到明确支持：

- 私域策略设计
- 生态协同设计
- OPC 相关结构化表达
- 多路径推演与取舍建议

### 4.2.5 缺少课后 30 天跟进模块

V1 明确提出课后跟进，目前项目未体现：

- 30 天跟进任务
- 双周案例推送
- 执行进展记录
- 方案再校准

### 4.2.6 知识库深度不够

当前仓库已有场景库、案例库、风险文档、模板文档，但与指导书要求相比还不足，特别缺少：

- 九要素问题库
- 每格常见方向建议库
- 行业商业终局案例
- 竞争力差异化模型知识
- 课程脚本库
- 更大规模结构化案例库

### 4.2.7 输出质量控制仍需强化

指导书强调“生成 -> 自检 -> 输出”，目前项目已在 LLM 报告部分加入部分校验逻辑，但还不够完整，主要缺：

- 全链路质量评分
- 无匹配内容时明确说明而非泛化表达
- 章节级可信度 / 来源说明
- 用户可见的回退说明与 warning 展示

### 4.2.8 课程现场使用能力仍需补齐

指导书强调 20-50 人并发、现场稳定可控，但当前项目主要偏本地 Demo，缺少：

- 并发压测结果
- 队列 / 限流 / 重试设计
- 课堂模式入口
- 讲师视角 / 学员视角分离
- 批量管理和课堂运营能力

## 5. 推进原则

后续推进建议遵循以下原则：

1. 先补“方法论流程缺口”，再补“知识深度”
2. 先保证课程现场可用，再扩展课后长期运营
3. 先做结构化引导能力，再做更重的智能生成能力
4. 模板兜底能力必须保留，不能只依赖 LLM
5. RAG 默认继续保持可选，不强行绑定主流程

## 6. 总体推进策略

建议分为四个阶段推进：

1. **阶段 A：夯实 Demo 到工作坊可用版本**
2. **阶段 B：补齐 V2 核心方法论流程**
3. **阶段 C：增强知识库、质量控制与交付稳定性**
4. **阶段 D：扩展课后跟进与课程运营能力**

这样的好处是：

- 能尽快把当前仓库提升到可用于课堂演示/试运行的水平
- 不会一开始就陷入大规模知识库建设和复杂平台化
- 可以边用边验证方法论是否真的需要产品化固化

## 7. 分阶段推进计划

## 7.1 阶段 A：夯实当前 Demo

### 目标

把当前项目从“开发完成的 Demo”推进到“可稳定演示和内部试用”的版本。

### 重点任务

#### A1. 配置与文档一致性治理

- 统一前端端口为 `3001`
- 修正 `.env.example`、`README.md`、旧说明文档中的过期信息
- 统一 RAG 路由说明，避免 `/api/rag` 与 `/rag` 混淆
- 清理启动脚本中的机器绑定说明

#### A2. 报告链路稳定性补强

- 完成 LLM 报告的回退验证
- 在前端展示 `used_llm`、`used_rag`、`warnings`
- 对生成失败、缺少前置数据、超时等情况做明确提示
- 补齐报告导出链路的操作说明

#### A3. 主流程测试补齐

- 已完成：覆盖问卷 -> 画像 -> 画布 -> 场景 -> 案例 -> 报告 主链路
- 已完成：覆盖模板模式与 LLM 回退模式
- 已完成：覆盖导出接口可用性
- 已完成：覆盖关键错误路径
- 已完成：覆盖 cases 自动补齐后再生成报告
- 已完成：覆盖 `/scenario-recommendations` 旧别名兼容
- 已完成：覆盖 `report_id` 不存在时报告详情 / Markdown / DOCX / 打印版统一返回 `404`
- 已完成：增加真 LLM 成功路径的可选集成测试开关，默认不运行，仅在本地同时配置 key、模型和 `RUN_LIVE_LLM_TESTS=true` 时启用

当前进展：

- 已新增并验证 `backend/tests/test_api_main_flow.py`
- 当前结果为 `7 passed, 1 skipped`
- 其中可选跳过项为真 LLM 成功路径测试，用于本地显式验证，不纳入默认回归集

### 阶段交付物

- 可用于稳定演示的版本
- 更新后的项目文档
- 主链路测试用例与关键错误路径回归基线
- Demo 演示脚本

### 验收标准

- 新环境按文档可在 30 分钟内完成启动
- 模板报告生成成功率高
- LLM 不可用时系统可稳定回退
- 导出功能可用
- A3 主流程测试默认可本地重复执行，真 LLM 成功路径仅在显式开关开启时运行

## 7.2 阶段 B：补齐 V2 核心方法论流程

### 目标

把系统从“自动生成结果”升级为“按方法论引导用户共同完成方案设计”的工具。

### 重点任务

#### B1. 引入课前输入与初始化导入

- 设计课前输入模板
- 增加文本 / Markdown / 表单导入能力
- 支持从非结构化描述中抽取关键信息
- 支持自动预填企业画像与部分画布格子

当前设计产物：

- `docs/B1_PRE_INPUT_IMPORT_DESIGN.md`：已完成 B1 首版方案设计，明确输入范围、数据分层、接口建议、预填边界、来源标记、前端交互与 MVP 验收标准

#### B2. 重构点阶段流程

- 新增“突破要素选择”步骤
- 支持从画布 9 要素中选 2-3 个突破点
- 针对每个突破点生成多条方向建议
- 在方向建议基础上再生成场景优先级矩阵

#### B3. 明确线阶段独立输出

- 新增“差异化竞争力设计”页面或模块
- 引入 VP 重构视图
- 引入系统串联型方案输出
- 从“点状场景列表”升级为“策略组合方案”

#### B4. 面阶段补齐商业终局模型

- 在报告结构中新增“商业终局”模块
- 将“私域 + 生态 + OPC”转成结构化模板
- 支持输出多个推进路径方案及利弊比较

### 阶段交付物

- V2 方法论流程原型
- 新的步骤式交互页面
- 突破要素与方向延展模块
- 商业终局结构化输出

### 验收标准

- 用户能明显感知“先选突破点，再延展方向，再出场景”
- 报告中能清晰看到“点 -> 线 -> 面”的演进逻辑
- 差异化竞争力与商业终局不再只是报告章节名，而成为独立可操作模块

## 7.3 阶段 C：知识库与质量控制增强

### 目标

让系统输出更像咨询结果，而不是一般性 AI 文案。

### 重点任务

#### C1. 扩充知识库结构

- 建立九要素问题库
- 建立要素方向建议库
- 扩充行业案例库
- 增加商业终局案例和框架库
- 增加竞争力差异化模型库
- 增加课程脚本库

#### C2. RAG 进入增强模式

- 按行业 -> 规模 -> 痛点建立分层检索策略
- 增加来源标注和引用摘要
- 优化 mock / real embedding 状态提示
- 明确哪些模块使用 RAG，哪些模块继续走规则或模板

#### C3. 建立质量护栏

- 为关键章节增加自检规则
- 明确“无匹配内容”输出规范
- 对 ROI、风险、案例适用性增加校验
- 为报告加入章节级 warning 或置信说明

### 阶段交付物

- 结构化知识库扩展版本
- RAG 增强版检索策略
- 报告质量检查规则清单
- 更接近咨询交付质量的输出版本

### 验收标准

- 报告关键结论能够说明依据
- 案例与行业相关性明显提升
- 用户可区分“规则结果”“知识库召回”“LLM 生成”

## 7.4 阶段 D：课后跟进与课程运营

### 目标

从“工作坊现场工具”拓展到“课程前中后全周期工具”。

#### D2. 双周案例推送与再校准

- 根据行业、场景、所选突破点推送案例
- 支持基于新案例重新评估方案
- 支持更新 90 天计划

#### D3. 讲师运营视角

- 讲师可查看学员推进状态
- 支持课堂分组和批量点评
- 支持学员成果汇总与导出

### 阶段交付物

- 课后跟进模块
- 再校准机制
- 讲师工作台雏形

### 验收标准

- 学员可在课后持续回访并更新方案
- 讲师可查看进度并进行点评

## 10. 风险与应对

### 风险 1：一开始做得过重

如果过早投入大规模知识库、复杂 RAG、完整平台化，会拖慢主流程验证。

应对：

- 先做 P0 / P1
- 优先验证 V2 流程中的“突破要素 -> 方向延展 -> 差异化设计”是否真正提升体验

### 风险 2：过度依赖 LLM

如果把关键输出完全交给 LLM，会带来稳定性和可控性问题。

应对：

- 保留模板兜底
- 对关键章节保持结构化生成 + 规则校验
- 只让 LLM 负责增强表达和归纳

### 风险 3：知识库建设投入大、见效慢

指导书中的知识库范围很广，如果一次性铺开，成本较高。

应对：

- 先围绕高频行业和核心场景建设
- 先补“九要素问题库 + 突破方向建议库 + 标杆案例”
- 后续再扩展行业终局库和脚本库

### 风险 4：课堂场景稳定性不足

如果没有经过并发和异常演练，课程现场风险较高。

应对：

- 阶段 A 就加入压测和异常演练
- 关键结果要有缓存或可重试机制
- 设计课堂演示模式和兜底操作路径

## 11. 结论

当前项目已经具备较好的技术底座，适合作为后续产品化推进的起点。

如果以 V1 为目标，当前项目已经完成了约 60% - 70% 的主链路雏形；
如果以 V2 为目标，当前项目更像完成了底层骨架，但关键的“方法论引导层”“商业终局层”“课后跟进层”仍需继续建设。

因此最合理的推进路径不是推翻重做，而是：

1. 先把当前 Demo 打磨成稳定可演示版本
2. 再补 V2 最关键的流程升级
3. 然后用知识库和质量护栏提升输出专业度
4. 最后扩展到课后跟进与讲师运营

整体建议：**以“工作坊可用”作为下一阶段最重要的短期目标，以“V2 方法论产品化”作为中期目标，以“课程前中后全周期运营”作为长期目标。**
